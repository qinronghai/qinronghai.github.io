创建对象

创建对象的两种方式

  1. 使用Object构造函数
  2. 使用字面量

工厂模式

用于抽象创建特定对象的过程。工厂意味着批量生产，工厂模式也意味着标准化地批量生产对象。

这种工厂模式虽然可以解决创建多个类似对象的问题，但没有解决对象标识问题（即新创建的对象是什么类型）

判断js对象类型的多种方法，附详细说明 - 掘金

使用工厂模式只知道实例是Object类型的，具体是什么类型的并不知道。

构造函数模式

用于创建特定类型对象的。像 Object 和 Array 这样的原生构造函数，运行时可以直接在执行环境中使用。当然也可以自定义构造函数，以函数的形式为自己的对象类型定义属性和方法。

相比于工厂模式:

  1. 没有显式的创建对象
  2. 属性和方法直接赋值给了this
  3. 没有显示的return

⚠️注意: 构造函数的函数名的首字母要大写。

用new操作符创建对象时发生的事情

  1. 在内存中创建一个新对象
  2. 将构造函数作用域赋给新的对象（即this指向新对象）
  3. 执行构造函数中的代码
  4. 返回新生成的对象实例

构造函数的问题

在构造函数中定义的方法会在每个实例上都创建一遍，对于起前面的person1 和 person2都有名为sayName()的方法，但这两个方法不是同一个Function实例。

这是因为在js中函数也是对象，因此在定义函数时，都会初始化一个对象，所以每个方法就不是同一个Function实例。

因为实例的sayName()方法都是做一样的事，所以没必要定义两个不同的Function实例。

解决这个问题

把函数定义转移到构造函数外部。

但这如果每个构造函数都这么用方法的话就需要在全局作用域中定义多个函数，这又会导致自定义类型引用的代码不能很好地聚集在一起。

原型模式

每个函数都会创建一个prototype属性，这个属性是一个对象。使用原型对象地好处事，在它上面定义的属性和方法可以被对象实例共享。这就解决了构造函数的问题。

理解原型

每个函数都有一个prototype属性引用（指向原型对象）。

所有原型对象都有一个constructor属性引用（指回与之关联的构造函数）。

在自定义构造函数时，原型对象默认只会获得 constructor 属性，其他的所有方法都继承自Object。

同一个构造函数创建的两个实例，共享同一个原型对象。

原型的层级

虽然可以通过实例读取原型对象上的值，但不可能通过实例重写这些值。

如果在实例上添加了一个与原型对象中同名的属性，那就会在实例上创建这个属性，这个属性会遮住原型对象上的属性。（就近原则）

即使在实例上把这个属性设置为 null，也不会恢复它和原型的联系。不过，使用 delete 操作符可以完全删除实例上的这个属性，从而让标识符解析过程能够继续搜索原型对象。

原型的问题

  1. 弱化了向构造函数传递初始化参数的能力，导致所有实列默认都取得相同的属性值。

  2. 原型的共享特性，即所有属性是在实例间共享的，这对于函数方法来说比较合适。问题在于引用值的属性。

    可以看到原型上的friends属性是一个字符串数组，实例person1往自己的friends数组中添加了一个值。这导致原型上同步添加，并且另一个实例person2的数组也同步添加了。这如果有意为之还可以，但是一般来说，不同的实例应该有属于自己的属性副本。所以，这就是实际开发中通常不单独使用原型模式的原因。
